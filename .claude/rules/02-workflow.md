# フェーズ詳細・コマンド一覧

> ### Claude Codeについて
> **Claude Code** はターミナル・VS Code・デスクトップアプリから起動できる。
> プロジェクトルートで `claude` コマンドを実行し、対話形式で操作する。
> `.claude/commands/` に置いたMarkdownが `/コマンド名` として使用できる。
> MCP サーバー（UnityMCP等）は `.claude/settings.json` で設定すれば CLI/VS Code/Desktop どこからでも使用可能。

## 1. 企画フェーズ
**使用ツール:** Claude Code CLI

1. ターミナルでプロジェクトルートに移動して `claude` を起動
2. `/new-spec` を実行
3. Claudeとゲームデザインの対話を行う（納得いくまで続けられる）
4. 「OK」「完成」「これでいく」と伝えると仕様書とPROJECT.mdが自動生成される
5. 内容を確認して「コミットして」と依頼

**対話の流れ：**
```
種を植える   → どんなアイデアでも受け取る（一言でもOK）
掘り下げる   → 感情体験・コアループ・ユニークさを1問ずつ深掘り
検証する     → まとめを提示して「このイメージで合ってる？」
スコープ確認 → MVPを定義して作れる規模に絞る
承認         → 企画者がOKを出したら仕様書生成
```

> ⚠️ Claudeは企画者の承認なしに仕様書を生成しない。

---

## 2. 設計フェーズ
**使用ツール:** Claude Code CLI

1. `/gen-design` を実行
2. ゲームデザイン設計を確認する（難易度曲線・フィードバックループ・バランス数値・MVPの定義）
3. 技術設計を確認する（クラス構成・依存関係・メソッド定義）
4. **UI設計を確認する**（レイアウト方針・カード構成・共通コンポーネント）
5. 人が設計書を確認して「コミットして」と依頼

> ⚠️ UI設計はシーン構築の前に確定させること。レイアウト方針（タブ/カラム/グリッド等）やカード構造（左画像+右テキスト等）が未定のままシーン構築に入ると、大幅な手戻りが発生する。

---

## 3. 実装フェーズ
**使用ツール:** Claude Code CLI + Claude Desktop（UnityMCP）

> ⚠️ このフローはモック制作が目的。サウンド・アニメーション等の演出は最低限でよい。

### 3a. コアループ実装（まず動かす）
1. `/gen-scripts core` を実行（コアループに必要なスクリプトのみ生成）
2. `/gen-scene core` を実行（UnityMCP経由でシーン構築）
3. Unityで実行して動作確認
4. 「コミットして」と依頼 → **プレイテストへ（3b）**

> ⚠️ シーン構築は **UnityMCP 経由** で行うこと。
> コードのみでのUI構築は視覚的フィードバックがなく、レイアウト崩れの原因になる。

### 3b. プレイテスト ← 重要
`/playtest` を実行して面白さを検証する。
- 面白い → 残りの実装へ（3c）
- 面白くない・仕様を変えたい → 設計フェーズへ戻る

### 3c. 残り実装（面白さが確認できたら）
1. `/gen-scripts full` を実行（残りのスクリプトを生成、core生成済みはスキップ）
2. `/gen-scene full` を実行（UnityMCP経由でシーン構築）
3. 人が全体を確認して「コミットして」と依頼

---

## 4. テストフェーズ
**使用ツール:** Claude Desktop（UnityMCP）+ Claude Code CLI

1. Claude Desktopを起動・UnityMCPと接続
2. Unityで実行しながら動作確認
3. `/playtest` で面白さを定期的に検証する

| 問題の種類 | 対応方法 |
|---|---|
| バグ（動作がおかしい） | `/debug` で修正 → 修正記録に残す |
| 数値バランスの問題 | GameBalanceSOの数値を調整 |
| 面白さの根本問題 | 設計フェーズへ戻る |
| 仕様の変更 | 仕様書を更新 → 設計フェーズへ戻る |

4. 定期的に `/check-diff` で実装とドキュメントの差分を確認
5. 仕様書更新のたびに「コミットして」と依頼

## Claude Code コマンド一覧

> **Claude Code（CLI）専用。** ターミナルでプロジェクトルートに移動後 `claude` を起動し、`/コマンド名` で実行。
> コマンドファイルは `.claude/commands/` に配置されており、Gitで管理される。

| コマンド | 使用フェーズ | 内容 |
|---|---|---|
| `/new-spec` | 企画 | 仕様書テンプレートを展開して対話形式で作成 |
| `/gen-design` | 設計 | 仕様書を読んで設計書を自動生成 |
| `/gen-scripts [core\|full]` | 実装 | 設計書からスクリプト生成（core: コアループのみ、full: 残り、引数なし: 全部） |
| `/gen-scene [core\|full]` | 実装 | シーン構築（core: 最小限、full: 残り、引数なし: 全部） |
| `/playtest` | 実装・テスト | 面白さを対話形式で検証・問題を分類して次の行動を提案 |
| `/check-diff` | テスト | 実装とドキュメントの差分をレポート形式で出力 |
| `/debug` | テスト | バグ調査・修正 |
| `/update-docs` | 共通 | 実装に合わせてDocs一式を更新・変更履歴を追記 |

### コマンドとツールの使い分け

```
企画フェーズ         → /new-spec
設計フェーズ         → /gen-design
スクリプト生成       → /gen-scripts core / full
シーン構築           → /gen-scene core / full（UnityMCP必須）
プレイテスト         → /playtest（UnityMCP推奨）
テストフェーズ       → /debug, /check-diff（UnityMCP推奨）
```

> **ツール使い分けの原則：**
> - **コード生成・ファイル操作・Git** → MCP不要（CLIの標準機能で十分）
> - **シーン構築・UI配置・動作確認** → UnityMCP必須（コードのみだとレイアウトが崩れる）

## .claude/agents/（現在は空）

プロジェクト固有の専門エージェントが必要になったら追加する。
初期は空。Claude Code標準のAgentツール（バックグラウンド実行）で十分な場合が多い。

> **追加の目安：** 同じ種類の作業を繰り返し委譲する場合、プロジェクト固有のパターン（フォント設定・レイアウトルール等）をエージェント定義に埋め込むと効率的。

## Commands / Skills 使い分け

### Commands（.claude/commands/）← 現在使用中
**人が意図的に起動するワークフロー全体に向く**

```
✅ フェーズの起点となる操作（/new-spec, /gen-design）
✅ 複数ステップを順番に踏む処理（/gen-scripts）
✅ 「今やる」タイミングが明確なもの（/playtest, /check-diff）
```

### Skills（.claude/skills/）← 将来の移行候補

```
✅ 参照ファイル（テンプレート・規約集）を一緒に持ちたい場合
✅ 手順が長くなりスクリプトも同梱したい場合
⚠️ 現在は commands で十分
```
