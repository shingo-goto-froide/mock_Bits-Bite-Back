# モック制作フロー

## フェーズ概要

```
企画 → 設計 → コアループ実装 → プレイテスト →┬→ 残り実装 → テスト
              ↑_______________|               │
              |（面白くない）                   ├→ 仕様更新 → 設計更新 → 残り実装
              ↑_______________________________|
                            （仕様追加・変更あり）
```

### プレイテスト後の判断
- **面白い＋仕様変更なし** → そのまま残り実装へ
- **面白い＋仕様追加/変更あり** → 仕様書更新 → `/gen-design` で設計更新 → 残り実装
- **面白くない** → 設計フェーズからやり直す。残り実装に進まない

### ドキュメント依存関係
```
仕様書 → 設計書 → スクリプト → シーン構成書 → シーン構築
```

---

## 各フェーズの進め方

| フェーズ | ツール | 種別 | 備考 |
|---|---|---|---|
| 企画 | `/new-spec` | skill | 対話形式で仕様書を作成 |
| 設計 | `/gen-design` | skill | ゲームデザイン + UI設計 + 技術設計 |
| コアループ実装 | gen-scripts → gen-scene | agent | シーン構築はUnityMCP必須 |
| プレイテスト | `/playtest` | skill | 面白さの検証。面白くなければ設計へ戻る |
| 残り実装 | gen-scripts → gen-scene | agent | 面白さが確認できてから |
| テスト | `/debug`, `/check-diff` | skill | バグ修正・差分確認 |
| ドキュメント更新 | update-docs | agent | 実装とドキュメントの同期 |

---

## skills と agents の使い分け

- **skills**（`/スキル名` で起動）：ユーザーとの対話が必要なもの。メインコンテキストで実行
- **agents**（自然言語 or @mention で起動）：投げっぱなしでOKなもの。別コンテキストで実行、メインを圧迫しない

---

## ツール使い分けの原則

- **コード生成・ファイル操作・Git** → MCP不要（CLIの標準機能で十分）
- **シーン構築・UI配置・動作確認** → UnityMCP必須（コードのみだとレイアウトが崩れる）

---

## ロール別：各フェーズで確認すること

### 企画者
- 設計: ゲームデザイン・UI設計を確認。技術設計はClaudeに任せてOK
- 実装: Unityで動かして「想定通り動くか」を確認。コードは見なくてよい
- テスト: 問題があれば症状を伝える。「〇〇ボタンを押しても反応しない」等
- `/check-diff` は使わなくてよい

### エンジニア
- 設計: 技術設計（クラス構成・依存関係）もレビュー
- 実装: コードレビュー・コンパイル確認
- テスト: `/debug` でエラーログから原因特定、`/check-diff` でコードと仕様の整合性確認
